04 — Questions

Frequently asked.

How can you audit a GTM container without access?

Google serves the gtm.js file publicly from googletagmanager.com — every browser that loads a site with GTM has to fetch it. We do the same. The container's full configuration (tags, variables, trigger conditions) is right there in that public script. No authentication needed.

Is this legal? Is it ethical?

Yes to both. The gtm.js file is intentionally public — Google designed it that way. No site is being scraped beyond what their server already sends to every visitor. We read the public configuration, we don't modify anything, we don't access any private data. Competitors looking at each other's public tracking has been standard practice for years.

What can't this tool see?

Server-side GTM containers (we detect their presence but not their internals), the runtime contents of Custom HTML tags (we see they exist, not what they execute), Single Page Apps that load GTM dynamically after initial render, and anything fired purely server-side from the host's own infrastructure.

Why does it show 153 tags but only list a few?

The summary section shows tags the AI flagged as notable — either passing key checks, warning conditions, or actively broken. The full container inventory expander lists every single tag in the container, including the ones that are just doing their job quietly.

How accurate is the AI audit?

The audit logic uses Llama 3.3 70B (via Groq) and is grounded in the actual parsed container data — not vibes. The model is given the real tag names, variable names, and trigger conditions extracted from gtm.js, plus a strict checklist of 2026 best practices to apply (UA deprecation, Consent Mode v2, snake_case events, Conversion Linker, etc.).

Can I export the audit?

Not yet — on the roadmap. For now, the full container inventory is in the page so you can copy-paste it. Tell me if you want PDF or CSV export and I'll prioritize it.
